## 1. 产品定位

**一句话**：说句话就能记账的微信小程序，你和伴侣的共同账本。

**痛点描述**：
- 现有记账小程序操作繁琐：选分类 → 选账户 → 输金额 → 确认，至少 4 步
- 不支持批量录入，多条支出要一条条记
- 情侣共账场景支持差，要么全公开要么全私密，没有"部分共享"

**解决方案**：
- 自然语言输入："午饭32，地铁4块，超市买水3.5，用支付宝" → AI 一次解析 3 条账单 → 确认卡片 → 一键入账
- 账户驱动可见性：花个人账户的钱进个人账本（仅自己可见），花共同账户的钱进共同账本（双向可见）

## 3. 账户体系

### 3.1 账户类型

| 账户类型 | 可见性 | 示例 | 余额计算 |
|----------|--------|------|----------|
| 个人账户 | 仅自己 | 支付宝、微信、现金、信用卡 | 期初 + 收入 - 支出 |
| 共同账户 | 双向可见 | 支付宝小荷包、共同存款 | 期初 + 收入 - 支出 |

### 3.2 账户管理
- 每个用户独立创建和维护自己的账户
- 新建账户时填写「当前余额」作为期初数据
- 系统自动生成一条「期初调整」记录
- 后续每笔记账自动增减账户余额
- 账户可随时编辑名称、修改余额（生成调整记录）

### 3.3 可见性规则

**核心规则：花哪里的钱 → 进哪个账本**

- 从个人账户支出 → 个人账本 → 仅自己可见
- 从共同账户支出 → 共同账本 → 双方可见
- 从共同账户收入 → 共同账本 → 双方可见
- 从个人账户收入 → 个人账本 → 仅自己可见

### 3.4 双人绑定流程

1. 用户 A 注册 → 系统生成邀请码
2. 用户 B 注册 → 输入 A 的邀请码 → 双方绑定关系
3. 绑定后共同账户出现，双方均可查看和记账
4. 可解绑，解绑后共同账本数据冻结归档

## 4. AI 对话录入（核心功能）

### 4.1 录入方式

| 方式 | 说明 | 示例 |
|------|------|------|
| 单条输入 | 一句话记一笔 | "午饭32，用微信" |
| 批量输入 | 多条一次解析 | "午饭32，地铁4块，超市买水3.5，用支付宝" |

### 4.2 AI 解析流程

```
用户输入自然语言
    ↓
调用 DeepSeek API（云函数中转）
    ↓
返回结构化数据数组：
[
  { description: "午饭", amount: 32, category: "餐饮", account: "微信", type: "支出" },
  { description: "地铁", amount: 4, category: "交通", account: "微信", type: "支出" },
  ...
]
    ↓
前端渲染确认卡片（可修改每条的金额/分类/账户）
    ↓
用户确认 → 批量写入数据库 + 扣减对应账户余额
```

### 4.3 智能默认逻辑

| 字段 | 默认规则 |
|------|----------|
| 金额 | AI 从文本提取数字 |
| 分类 | AI 根据消费内容自动归类 |
| 账户 | 用户未指定时，默认使用上一次的账户 |
| 类型 | 默认支出，输入"工资/退款/红包"等关键词时识别为收入 |
| 日期 | 默认今天，支持"昨天/前天"等时间词 |

### 4.4 AI 模型选型

| 项目 | 选择 |
|------|------|
| 模型 | DeepSeek API |
| 每条成本 | ~¥0.001 |
| 月成本（每天20条） | ~¥0.6 |
| 优势 | 国内API低延迟、识别准确、支持批量解析、成本极低 |
| Prompt 设计 | 系统提示词包含：用户自定义分类列表、账户列表、输出JSON格式要求 |

### 4.5 AI Prompt 设计要点

```
你是记账助手。解析用户输入的消费记录，返回JSON数组。
规则：
1. 从自然语言中提取：描述、金额、分类、支付账户、收支类型
2. 分类从以下列表中选择：[用户自定义分类列表]
3. 账户从以下列表中选择：[用户账户列表]
4. 用户未指定账户时，使用上次账户：[上次账户名]
5. 支出/收入：默认支出，"工资/退款/红包/转账收入"为收入
输出格式：[{ "description": "", "amount": 0, "category": "", "account": "", "type": "expense/income", "date": "YYYY-MM-DD" }]
```

## 7. 数据流（完整链路）

```
用户输入自然语言
    ↓
[云函数] parseTransaction(text, userId)
    ↓ 调用 DeepSeek API
    ↓ 传入：用户分类列表、账户列表、上次账户
    ↓ 返回：结构化 JSON 数组
    ↓
前端渲染确认卡片
    ├── 每条记录可编辑：金额、分类、账户、类型、日期
    ├── 可删除某条
    └── 可添加遗漏的条目
    ↓
用户点击「确认入账」
    ↓
[云函数] saveTransactions(records, userId)
    ├── 批量写入 transactions 集合
    ├── 批量更新 accounts 集合余额
    └── 批量更新 budget 统计
    ↓
完成，返回更新后的账户余额
```

## 8. 技术架构

### 8.1 技术选型

| 层 | 技术 | 说明 |
|----|------|------|
| 前端 | 微信小程序原生（WXML + WXSS + JS） | 不用框架，原生最稳 |
| 后端 | 腾讯云开发 Cloudbase | 云函数 + 云数据库 + 云存储 |
| 数据库 | Cloudbase 内置数据库（文档型/MongoDB） | 免运维，免备案 |
| AI | DeepSeek API | 云函数内调用，前端不直接调 |
| 登录 | 微信登录（wx.login） | 无需额外认证 |

### 8.2 架构隔离设计（为未来迁移自建服务器准备）

1. **DAO 层隔离**：所有数据库操作封装在 DAO 模块，迁移时只改 DAO 层
2. **云函数无状态**：业务逻辑写成纯函数，迁移时直接搬到 Express/Koa 路由
3. **API 层统一封装**：前端所有请求走 `api.js` 统一封装，迁移时改一个文件
4. **数据结构对齐**：Cloudbase Collection 字段与未来 MySQL Table 字段保持一致

### 8.3 数据库集合设计（Cloudbase Collections）

```
users          - 用户信息
  { _id, openid, nickName, avatar, inviteCode, partnerId, createdAt }

accounts       - 账户
  { _id, userId, name, type: "personal/shared", balance, initialBalance, sortOrder, createdAt }

categories     - 分类
  { _id, userId, name, type: "expense/income", isDefault, sortOrder }

transactions   - 交易记录
  { _id, userId, description, amount, category, accountId, accountType: "personal/shared",
    type: "expense/income", date, createdAt }

budgets        - 预算
  { _id, userId, scope: "personal/shared", month, amount, createdAt }
```

### 8.4 迁移路径

| 阶段 | 用户量 | 方案 | 月成本 | 迁移动作 |
|------|--------|------|--------|----------|
| 内测期 | 2-50人 | Cloudbase 免费额度 | ¥0~1 | 无 |
| 增长期 | 500-2000人 | Cloudbase 按量付费 | ¥20-80 | 无 |
| 成熟期 | 5000+人 | 自建服务器 | ~¥200 | DAO层迁移 + 数据导入 + 域名备案 |

迁移工作量预估：2-3 天（前提是前期架构隔离做好）

## 10. 微信小程序上架注意事项

### 10.1 主体选择

| 维度 | 个人主体 | 企业主体 |
|------|----------|----------|
| 注册门槛 | 身份证 | 营业执照 |
| 能否收费 | 不能 | 能 |
| 能否接广告 | 能（流量主） | 能 |
| 能否开通支付 | 不能 | 能 |
| 类目限制 | 部分受限 | 基本全开放 |
| 适用阶段 | 内测 + 早期 | 商业化阶段 |

**结论**：前期个人主体上架，M6+ 转企业主体。

### 10.2 备案说明

- Cloudbase 方案：**不需要域名备案**，省去 2-3 周流程
- 自建服务器方案：需要域名 ICP 备案（2-3 周）
- 小程序本身不需要单独 ICP 备案

### 10.3 记账类目审核
- 「记账」属于工具类，个人主体可注册
- 审核相对宽松，注意不要涉及金融理财功能

## 11. 一期 Non-goals（明确不做）

| 不做项 | 原因 |
|--------|------|
| 投资理财追踪 | 超出记账范畴，一期聚焦 |
| 多币种汇率转换 | 复杂度高，需求小 |
| 发票/小票 OCR | AI 解析已够用，OCR 增加成本 |
| 微信/支付宝账单自动导入 | 需要授权对接，合规复杂 |
| 家庭多成员（>2人） | 一期聚焦情侣双人场景 |
| Siri/语音输入 | 二期做 |
| 分类预算 | 一期仅总预算 |
| 预算趋势分析 | 二期做 |

**原则：先把"说话就能记账"这一件事做到极致，不铺功能。**
